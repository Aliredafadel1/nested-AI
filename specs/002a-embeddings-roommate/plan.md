# Implementation Plan: Phase 2a — Embeddings + Roommate Matching

**Branch**: `002a-embeddings-roommate` | **Date**: 2026-06-06 | **Spec**: specs/002a-embeddings-roommate/spec.md

## Summary

Activate the BGE-M3 embedding pipeline for listings and student profiles, and build the roommate compatibility engine. All 5 clarification decisions are locked (see spec). Core work: replace the `core/embeddings.py` stub with a full implementation, add two Celery task files (one per module), write a DB migration adding 5 `dim_*` vector columns to `student_profiles`, implement the roommate module's 5-file layout, and wire up `GET /roommate/matches` + `POST /roommate/requests`.

## Technical Context

**Language/Version**: Python 3.11

**Primary Dependencies**: `sentence-transformers==3.0.1`, `torch==2.3.1` (BGE-M3 runtime), `celery==5.4.0`, `redis-py==5.0.6`, `pgvector==0.3.0`, `sqlalchemy==2.0.30` async, `asyncpg==0.29.0`

**Storage**: PostgreSQL 16 + pgvector (1024-dim HNSW), Redis 7 (embed cache 48h TTL, distributed lock 30s TTL, LLM cache)

**Testing**: pytest + pytest-asyncio against real containerised PostgreSQL + Redis. No mocks.

**Target Platform**: Docker Compose — `worker-low` service (BGE-M3, 1GB memory cap), `api` service (FastAPI endpoints)

**Project Type**: Background task pipeline (Celery) + REST API additions to existing modular monolith

**Performance Goals**: Embed task completes within 5s per listing (BGE-M3 local inference). Match query returns within 500ms for ≤1000 candidates (pgvector ANN).

**Constraints**: BGE-M3 loads ONCE at `worker_init` — never per task. All vectors 1024-dim. Retry: 3 attempts, exponential backoff (10s → 30s → 90s).

**Scale/Scope**: 50 seed listings, ~5 demo students. pgvector HNSW handles millions — no scaling concern for Phase 2a.

## Constitution Check

- ✅ Module boundaries: `modules/housing/tasks.py` and `modules/users/tasks.py` — tasks call their own module's `service.py` only. No cross-module repository imports.
- ✅ Security: No new auth surface. LLM inputs: embed text goes through BGE-M3 locally, not an external API — prompt injection sanitizer not required for local models.
- ✅ AI cost discipline: BGE-M3 is free (local). No OpenAI/Anthropic calls in Phase 2a.
- ✅ No WebSockets introduced.
- ✅ Real DB in tests — no mocks.
- ✅ Redis keys via `RedisKeys.*` only.
